1. UNIT TESTING
================================================================================

DEFINITION:
Unit Testing is a software testing method where individual units or components
of a software application are tested in isolation to validate that each unit
performs as designed.

PURPOSE:
• Test individual functions, methods, or classes in isolation
• Catch defects early in the development process
• Ensure code quality and reliability
• Make code refactoring safer by detecting regressions
• Improve code design and maintainability
• Reduce debugging time and cost

KEY CHARACTERISTICS:
• Tests small, isolated units of code (functions, methods, classes)
• Uses mock objects or stubs for dependencies
• Tests both positive and negative scenarios
• Should be fast and independent
• Focuses on logic correctness
• Tests written and executed by developers
• Usually automated

BENEFITS:
✓ Early detection of bugs in development phase
✓ Easier to maintain and modify code
✓ Improves code quality and design
✓ Speeds up development process
✓ Reduces cost of fixing bugs
✓ Increases developer confidence
✓ Documents code behavior

SCOPE FOR RIDEBAZZAR:
Functions to Unit Test:
• User authentication (login, registration, validation)
• Password hashing and verification
• Car price calculations
• Booking date validation
• Payment amount calculations
• Search and filter algorithms
• Email/SMS notification formatting
• Data validation functions
• Role-based access control

UNIT TESTS EXAMPLES:

1. User Authentication Test:
   • Test valid login credentials
   • Test invalid username
   • Test invalid password
   • Test password hash verification
   • Test token generation

2. Car Management Test:
   • Test car creation with valid data
   • Test car creation with missing fields
   • Test price validation (positive numbers)
   • Test car status update (available/booked)
   • Test car deletion

3. Booking Test:
   • Test booking creation with valid data
   • Test date validation (future dates only)
   • Test booking amount calculation
   • Test booking status updates
   • Test booking cancellation

4. Payment Test:
   • Test amount validation
   • Test payment method validation
   • Test transaction ID generation
   • Test payment status tracking

TOOLS & FRAMEWORKS:
Frontend:
• Jest (JavaScript testing framework)
• Mocha (Test runner)
• Chai (Assertion library)
• Sinon (Mocking, stubbing)

Backend (Node.js):
• Jest
• Mocha
• Chai
• Sinon
• SuperTest (HTTP assertion library)

BEST PRACTICES:
1. Write tests before or alongside code (TDD)
2. Test one thing per test case
3. Use meaningful test names
4. Keep tests simple and readable
5. Avoid testing implementation details
6. Use setup and teardown methods
7. Aim for high code coverage (70-80%)
8. Keep tests independent
9. Mock external dependencies
10. Run tests frequently

EXAMPLE UNIT TEST CODE:

// User Login Test (JavaScript - Jest)
describe('User Authentication', () => {
  test('should login with valid credentials', async () => {
    const result = await loginUser('admin', 'admin123');
    expect(result.success).toBe(true);
    expect(result.user.username).toBe('admin');
  });

  test('should fail with invalid password', async () => {
    const result = await loginUser('admin', 'wrongpassword');
    expect(result.success).toBe(false);
    expect(result.message).toContain('Invalid password');
  });

  test('should fail with non-existent user', async () => {
    const result = await loginUser('nonexistent', 'password');
    expect(result.success).toBe(false);
    expect(result.message).toContain('User not found');
  });
});

// Car Creation Test

2. INTEGRATION TESTING
================================================================================

DEFINITION:
Integration Testing is a level of software testing where individual units
are combined and tested as a group. It verifies that different modules or
services work together as expected.

PURPOSE:
• Test interactions between integrated modules/components
• Verify data flow across different parts of the application
• Ensure system components communicate correctly
• Validate end-to-end workflows
• Identify issues that occur only when modules work together
• Test real database and external service interactions

KEY CHARACTERISTICS:
• Tests multiple components/modules together
• Uses real or near-real databases
• Tests actual external API calls
• Tests communication between frontend and backend
• Tests data integrity across components
• More complex than unit tests
• Takes longer to execute
• Requires integration test environment

BENEFITS:
✓ Detects integration defects
✓ Tests real-world scenarios
✓ Validates data flow between modules
✓ Ensures API contracts are met
✓ Tests database operations
✓ Identifies environment-related issues
✓ Increases confidence in system reliability

Black Box Testing is a testing method where the tester evaluates the software
without knowing its internal structure, design, or implementation. The focus
is on input and output.

PURPOSE:
• Test application from end-user perspective
• Validate functional requirements
• Ensure user workflows work correctly
• Test without knowledge of code
• Find real-world defects
• Verify user experience
• Test error handling and edge cases

KEY CHARACTERISTICS:
• Tests application as a whole
• No knowledge of internal code/architecture
• Focuses on functionality and user experience
• Tests valid and invalid inputs
• Tests boundary conditions
• Tests error messages
• Can be performed by non-technical testers
• Uses requirement specifications as basis

BENEFITS:
✓ Tests from user perspective

6. AUTHENTICATION & AUTHORIZATION SYSTEM
================================================================================

OVERVIEW:
RideBazzar implements a secure authentication and authorization system using
JWT (JSON Web Tokens) for stateless authentication and role-based access control
(RBAC) for authorization. This ensures only authorized users can access
specific features and data.

JWT TOKEN SYSTEM:

What is JWT?
JWT (JSON Web Tokens) is a method to securely transmit information as a JSON
object. Each token is digitally signed, ensuring it cannot be tampered with.

JWT Structure:
A JWT token consists of three parts separated by dots:
  header.payload.signature

1. Header: Contains token type and algorithm
   Example: { "alg": "HS256", "typ": "JWT" }

2. Payload: Contains claims (user data)
   Example: { "userId": 42, "username": "john", "role": "seller" }

3. Signature: Digital signature to verify token authenticity
   Created using: HMAC(header.payload, SECRET_KEY)

Example Token:
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.
eyJ1c2VySWQiOjQyLCJ1c2VybmFtZSI6ImpvaG4ifQ.
4Hs_Xdp-H7n9Km2_R8vJ_Q1pL2nO9M5jX3yZ

TOKEN GENERATION (During Login):

Step 1: User submits credentials
  POST /api/auth/login
  Body: { username: "john", password: "pass123" }

Step 2: Backend validates credentials
  - Retrieve user from database
  - Verify password using bcrypt.compare()
  - If invalid: return 401 Unauthorized

Step 3: Generate JWT token
  Code:
  const token = jwt.sign(
    {
      userId: user.id,
      username: user.username,
      role: user.role        // 'admin', 'seller', or 'user'
    },
    process.env.JWT_SECRET,
    { expiresIn: '24h' }     // Token expires in 24 hours
  );

Step 4: Send token to frontend
  Response: {
    success: true,
    token: eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...,
    user: { id: 42, username: "john", role: "seller" }
  }

TOKEN STORAGE (Frontend):

```javascript
// Store token in localStorage
localStorage.setItem('authToken', token);

// Every subsequent request includes token
const headers = {
  'Authorization': `Bearer ${localStorage.getItem('authToken')}`
};

// Example: Fetch with token
fetch('/api/cars/my-cars', {
  method: 'GET',
  headers: {
    'Authorization': `Bearer ${token}`,
    'Content-Type': 'application/json'
  }
});
```

Token Lifecycle Diagram:

1. LOGIN
   User enters credentials
        ↓
   Backend validates password
        ↓
   JWT token generated (valid for 24 hours)
        ↓
   Token sent to frontend
        ↓

2. STORAGE
   Token stored in localStorage
        ↓

3. USAGE
   Every request includes token in header:
   Authorization: Bearer <token>
        ↓

4. VALIDATION
   Backend receives request
        ↓
   Extracts token from header
        ↓
   Verifies signature (prevents tampering)
        ↓
   Checks expiration (still valid?)
        ↓
   Extracts user info (userId, role)
        ↓
   Adds to request object
        ↓

5. AUTHORIZATION
   Checks user role matches endpoint requirements
   Checks user owns resource (if needed)
   Grants or denies access
        ↓

6. RESPONSE
   Sends response or 403 Forbidden
        ↓

7. EXPIRATION
   After 24 hours, token no longer valid
   User automatically logged out
   Must login again to get new token

Security Considerations:
✓ Never send passwords in response
✓ Always validate token on backend
✓ Don't trust user role from frontend (send in header but verify)
✓ Use HTTPS to prevent token interception
✓ Set secure HttpOnly cookie if using cookies
✓ Implement logout to clear token from frontend
✓ Rate limit login attempts
✓ Hash passwords before storing
✓ Never expose JWT secret

8. COCOMO MODEL - PROJECT ESTIMATION & COST ANALYSIS
================================================================================

OVERVIEW:
The COCOMO (Constructive Cost Model) is a software project estimation model
used to predict effort, cost, schedule, and quality. This section applies COCOMO
to RideBazzar project for realistic planning and resource allocation.

WHAT IS COCOMO?

COCOMO is an algorithmic cost model that estimates:
• Development effort (in person-months)
• Project timeline (in months)
• Total project cost (in currency)
• Required team size
• Risk areas and critical paths

Based on:
• Project size (Lines of Code - LOC)
• Project type/complexity
• Cost drivers (team experience, complexity, tools, etc.)
• Historical data from similar projects

WHY USE COCOMO?

Benefits:
✓ Objective estimation (data-driven, not guesswork)
✓ Early identification of risks
✓ Better resource planning
✓ Cost justification to stakeholders
✓ Progress tracking against estimates
✓ Identifies schedule impacts early

COCOMO LEVELS:

Level 1: BASIC COCOMO
- Simple, high-level estimation
- Uses overall project size
- Suitable for early planning phases
- Less accurate but quick

Formula: Effort = a × (KLOC)^b
Where:
  KLOC = thousands of lines of code
  a = 2.4 (basic multiplier)
  b = 1.05 (exponential factor)

Example:
  Project size = 15 KLOC
  Effort = 2.4 × (15)^1.05 = 37 person-months

Level 2: INTERMEDIATE COCOMO
- More detailed than Basic
- Considers 15 cost drivers
- Suitable for project planning
- Better accuracy

Cost Drivers Include:
1. Required reliability
2. Database size
3. Product complexity
4. Execution time constraints
5. Main storage constraints
6. Virtual machine volatility
7. Computer turnaround time
8. Analyst capability
9. Applications experience
10. Systems programmer capability
11. Programming language experience
12. Software tools
13. Modern programming practices
14. Schedule constraints
15. Required development facilities

Formula: Effort = a × (KLOC)^b × EAF
Where:
  EAF = Effort Adjustment Factor
  Values range from 0.9 to 1.4 depending on cost drivers

Example:
  Base effort = 37 person-months
  EAF = 1.15 (medium-high complexity)
  Adjusted effort = 37 × 1.15 = 42.5 person-months

Level 3: DETAILED COCOMO
- Highly detailed estimation
- Project broken into modules
- Each module estimated separately
- Most accurate but very time-consuming
- Suitable for critical projects

Process:
1. Decompose project into modules
2. Estimate each module separately
3. Consider module interactions
4. Aggregate module estimates
5. Add overhead/integration effort

```javascript
      message: 'Missing required fields' 
    });
  }
  
  // 4. Validate payment method
  const validMethods = ['card', 'upi', 'netbanking', 'wallet'];
  if (!validMethods.includes(paymentMethod)) {
    return res.status(400).json({ 
      success: false, 
      message: 'Invalid payment method' 
    });
  }
  
  // 5. Fetch booking from database
  const [bookings] = await db.execute(
    'SELECT * FROM bookings WHERE id = ? AND buyer_id = ?',
    [bookingId, req.user.userId]
  );
  
  if (bookings.length === 0) {
    return res.status(404).json({ 
      success: false, 
      message: 'Booking not found' 
    });
  }
  
  // 6. Verify amount matches
  const booking = bookings[0];
  if (parseFloat(amount) !== parseFloat(booking.booking_amount)) {
    return res.status(400).json({ 
      success: false, 
      message: 'Amount mismatch' 
    });
  }
  
  // All validation passed, proceed to payment processing
});
```

Step 10: DATABASE TRANSACTION
```javascript
// Start database transaction for atomicity
const connection = await db.getConnection();
try {
  await connection.beginTransaction();
  
  // 1. Create transaction record
  await connection.execute(
    `INSERT INTO transactions 
     (booking_id, user_id, amount, payment_method, 
      payment_status, transaction_id, gateway_response, created_at)
     VALUES (?, ?, ?, ?, ?, ?, ?, NOW())`,
    [
      bookingId,
      req.user.userId,
      amount,
      paymentMethod,
      paymentStatus, // 'success' or 'failed'
      transactionId,
      JSON.stringify(gatewayResponse)
    ]
  );
  
  // 2. If payment successful, update booking
  if (paymentStatus === 'success') {
    await connection.execute(
      `UPDATE bookings 
       SET status = 'confirmed', payment_status = 'paid'
       WHERE id = ?`,
      [bookingId]
    );
    
    // 3. Update car availability
    await connection.execute(
      `UPDATE cars 
       SET status = 'booked'
       WHERE id = ?`,
      [booking.car_id]
    );
    
    // 4. Update seller stats
    await connection.execute(
      `UPDATE users 
       SET total_sales = total_sales + 1
       WHERE id = ?`,
      [booking.seller_id]
    );
  }
  
  // Commit all changes together
  await connection.commit();
  
} catch (error) {
  // Rollback if any error
  await connection.rollback();
  throw error;
} finally {
  connection.release();
}
```

COMPREHENSIVE APPROACH AND SYNERGY:

Unit Testing forms the foundation of quality by catching defects at the earliest possible stage when they are least expensive to fix. By testing individual functions, methods, and classes in isolation, developers gain immediate feedback about the correctness of their implementation and can refactor confidently knowing that tests will catch any unintended side effects. Integration Testing bridges the gap between unit testing and black box testing by validating that components work correctly when combined. This layer of testing is critical because defects often arise not from individual components but from the interactions between them, from data flow inconsistencies, and from assumptions about component behavior that don't hold in the integrated system. Integration tests validate both technical contracts between components and business processes that depend on multiple systems working in concert. Black Box Testing completes the quality validation by testing the system from the user's perspective, ensuring that the product actually delivers the desired user experience and functionality without requiring knowledge of the internal implementation.
